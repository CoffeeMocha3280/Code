# 统一可执行程序说明

这份说明文档用于解释当前“统一成一个可执行程序”的整理结果，方便组员快速了解：

- 哪些文件是最终主线会用到的
- 哪些旧文件被保留但不再是主线
- 这次为了整合做了哪些改动

## 1. 最终主线使用到的文件

- `Exp1_Lexer.c`
  - 实验一主实现
  - 保留原有双缓冲词法分析逻辑
  - 现在除了能给实验二提供 `yylex()`，还额外提供了统一主程序调用的接口

- `Exp1_LexerAPI.h`
  - 新增
  - 声明实验一对外提供的接口函数

- `Exp2_Parser.y`
  - 实验二主实现
  - 保留原有 Yacc 文法、语法树构造、错误处理、归约序列记录逻辑
  - 现在额外提供“按文件分析、返回语法树、打印结果、释放语法树”的接口

- `Exp2_SyntaxTree.h`
  - 新增
  - 抽出实验二语法树结点 `Node` 的公共定义
  - 让实验二和实验三都能使用同一份语法树结构

- `Exp2_ParserAPI.h`
  - 新增
  - 声明实验二对外提供的接口函数

- `Exp3_CodeGen.c`
  - 新增
  - 实验三主实现
  - 不再自己做词法分析和语法分析
  - 只接收实验二生成的语法树，再输出三地址码

- `Exp3_CodeGen.h`
  - 新增
  - 声明实验三代码生成接口

- `Main.c`
  - 新增
  - 单独的主函数文件，也是统一程序的唯一总入口
  - 把实验一、实验二、实验三按固定顺序串成一个程序
  - 启动后会依次执行三个实验，不再让用户选择执行哪一个
  - 交互模式下会自动扫描当前目录中的 `txt` 文件，以及 `input_txt` 子目录中的 `txt` 文件，供用户直接选择

- `BuildUnified.bat`
  - 新增
  - 一键生成 `Exp2_Parser.tab.c` 并编译统一程序

- `Exp2_Parser.tab.c`
- `Exp2_Parser.tab.h`
  - 自动生成文件
  - 由 `Exp2_Parser.y` 通过 `win_bison` 生成
  - 统一程序实际编译时会使用到

## 2. 当前目录中各文件的作用

下面按“现在是否真正参与统一主线”来说明当前文件的作用。

### 2.1 主线必须保留的文件

- `Main.c`
  - 统一程序总入口
  - 负责扫描当前目录和 `input_txt` 子目录中的 `txt` 文件
  - 负责让用户选择输入文件或手动输入
  - 负责按固定顺序调用实验一、实验二、实验三

- `Exp1_Lexer.c`
  - 实验一词法分析主程序
  - 负责把源程序切分成 token
  - 统一程序运行时，实验一的输出就来自它
  - 同时也给实验二提供 `yylex()` 词法接口

- `Exp1_LexerAPI.h`
  - 实验一的接口声明文件
  - 让 `Main.c` 和实验二能够调用实验一暴露出来的函数

- `Exp2_Parser.y`
  - 实验二语法分析主文法文件
  - 使用 Yacc/Bison 进行语法分析
  - 在归约过程中构造语法树
  - 统一程序里的实验二功能主要由它决定

- `Exp2_ParserAPI.h`
  - 实验二的接口声明文件
  - 声明“分析文件、返回语法树、打印语法树、释放语法树”等函数

- `Exp2_SyntaxTree.h`
  - 实验二和实验三共用的语法树结点定义
  - 没有它，实验三就无法识别实验二生成的树结构

- `Exp3_CodeGen.c`
  - 现在真正的实验三实现文件
  - 不重新做语法分析
  - 直接读取实验二生成的语法树，并递归遍历生成三地址码

- `Exp3_CodeGen.h`
  - 实验三的接口声明文件
  - 让主程序可以调用实验三的三地址码生成函数

- `BuildUnified.bat`
  - 统一程序的一键构建脚本
  - 先调用 `win_bison` 把 `Exp2_Parser.y` 生成成 `Exp2_Parser.tab.c/.h`
  - 再调用 `gcc` 编译出统一可执行文件

- `UnifiedCompiler.exe`
  - 当前编译好的统一可执行程序
  - 直接运行它，就会顺序执行实验一、实验二、实验三

- `统一可执行程序说明.md`
  - 当前这份说明文档
  - 用来帮助组员理解现在的工程结构和各文件作用

- `input_txt`
  - 输入样例文件夹
  - 放实验测试用的 `txt` 输入文件
  - 统一程序启动后会自动扫描这个目录中的 `txt`

### 2.2 自动生成但有用的文件

- `Exp2_Parser.tab.c`
  - 由 `Exp2_Parser.y` 自动生成的 C 源文件
  - 是 Bison 生成出来的语法分析器实现
  - 编译统一程序时会被真正用到

- `Exp2_Parser.tab.h`
  - 由 `Exp2_Parser.y` 自动生成的头文件
  - 里面有 token 编号等定义
  - 实验一词法分析器在 parser 模式下会引用它

说明：

- 这两个文件不是手工维护的核心源码
- 如果删掉，也可以通过 `.\BuildUnified.bat` 重新生成

### 2.3 保留参考，但当前统一主线没有使用到的文件

- `Exp3_Parser.y`
  - 这是早期的过渡方案文件
  - 它代表“实验三自己再带一套 Yacc 文法和语法分析过程”
  - 现在最终主线已经改成“实验二生成语法树，实验三直接读取语法树生成三地址码”
  - 所以它目前只是参考文件，不参与统一程序编译和运行

### 2.4 输入测试文件的作用

- 当前目录下的 `txt` 文件
  - 可以作为统一程序的输入文件
  - 程序启动后会自动扫描并列出来

- `input_txt` 子目录下的 `txt` 文件
  - 也是统一程序的输入文件
  - 适合集中存放实验测试样例
  - 程序会以相对路径形式显示，例如 `input_txt/sample_input.txt`

## 3. 这次改动过的现有文件

- `Exp1_Lexer.c`
  - 新增了 `Exp1_ResetLexerState()`
  - 新增了 `Exp1_RunLexerFromFile()`
  - 把原来重复的词法输出循环整理成了可复用函数
  - 没有改动原有 token 识别规则

- `Exp2_Parser.y`
  - 把 `Node` 定义抽到了 `Exp2_SyntaxTree.h`
  - 新增了实验二对外接口函数
  - 把原来的 `main()` 保留为“独立运行模式”
  - 增加了宏 `EXP2_DISABLE_STANDALONE_MAIN`，用于统一程序编译时关闭独立 `main`
  - 没有改动原有文法和主要语法分析逻辑

## 4. 当前统一程序的工作方式

统一程序 `UnifiedCompiler.exe` 的流程如下：

1. 先执行实验一
   - 输出词法分析结果

2. 再执行实验二
   - 输出语法树和归约序列

3. 最后执行实验三
   - 使用实验二已经生成的语法树输出三地址码

## 5. 当前推荐运行方式

先重新构建：

```powershell
.\BuildUnified.bat
```

你可以有两种使用方式。

第一种：直接启动程序，进入交互模式，然后选择：

- 当前目录下某个 `txt` 文件
- 或 `input_txt` 子目录下某个 `txt` 文件
- 或手动输入源程序

注意：

- 这里不需要再选择执行哪个实验
- 输入来源确定后，程序会自动依次执行实验一、实验二、实验三

第二种：显式传入输入文件运行：

```powershell
.\UnifiedCompiler.exe your_input.txt
```

## 6. 一句话总结

现在的统一主线已经调整为：

`实验一词法分析 -> 实验二语法分析并构造语法树 -> 实验三根据语法树生成三地址码`

这满足了“最后统一成一个可执行程序”的要求，同时尽量保留了实验一和实验二原有的主要工作逻辑。
